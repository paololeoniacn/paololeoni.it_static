# Architettura V2: CV Orchestrator & Static Deployment

Questa documentazione definisce l'architettura **V2** del progetto `paololeoni.it`, basata sul paradigma dell'**AI Orchestrator Dashboard**.

---

## 1. Architettura del Sistema
A differenza della V1 (statica manuale), la V2 separa nettamente le responsabilità:

*   **ADMIN (CMS/Orchestrator)**: Un'interfaccia interattiva (`/admin/index.html`) alimentata da un server locale Python (`server.py`). Permette la modifica visuale di ogni sezione del CV e il salvataggio diretto nel database JSON.
*   **PUBLIC (Front-End)**: Il sito web finale (`/public/index.html`) che consuma dinamicamente il file `resumeData.json`. Utilizza CSS moderno con Glassmorphism e animazioni Neon.
*   **DATA (Source of Truth)**: Il file `data/resumeData.json` è l'unico punto di verità. Tutte le modifiche fatte nell'Admin si riflettono istantaneamente sul sito.

---

## 2. Workflow di Sviluppo e Aggiornamento
Per aggiornare il tuo profilo, segui questo "Protocollo Cerniera":

1.  **Avvio Locale**:
    ```bash
    ./handle_project.sh run
    ```
    Apri [http://localhost:8081/admin/](http://localhost:8081/admin/).

2.  **Editing & Sync**:
    - Modifica le sezioni (Lavoro, Skills, Certificazioni).
    - La preview a destra mostra i cambiamenti in tempo reale (Stile Premium).
    - Clicca **"Save to JSON"** per rendere permanenti le modifiche nel file locale.

3.  **Controllo Versione**:
    ```bash
    git add data/resumeData.json
    git commit -m "update: aggiunto nuovo progetto AI"
    git push origin main
    ```

---

## 3. Deploy su Cloudflare Pages
Il progetto è configurato per il deploy come sito statico su Cloudflare Pages. Grazie alla sua natura "dynamic-static", non è necessario buildare nulla (NPM non è richiesto per la produzione).

### Configurazione Wrangler (`wrangler.toml`)
Il file deve puntare alla root del progetto per garantire che il path relativo tra `public/` e `data/` sia mantenuto:

```toml
name = "paololeoni-it"
pages_build_output_dir = "." # Mantiene l'integrità dei path relativi
```

### Comando di Deploy
È stato automatizzato nell'helper di progetto:

```bash
# Esegue il deploy della cartella corrente su Cloudflare
./handle_project.sh deploy
```

---

## 4. Note sulla Sicurezza
- **Admin**: L'area `/admin/` è intesa per uso locale. Quando il sito è deployato su Cloudflare, l'area Admin sarà visibile ma il tasto "Save" non funzionerà (poiché Cloudflare Pages non supporta scritture dirette su file system in runtime).
- **Update Prod**: Per aggiornare la produzione, segui sempre il workflow **Local Admin -> Git Push**. Se desideri un Admin online funzionante, sarà necessario implementare una Cloudflare Worker con D1 (V3 Roadmap).

---

## 5. Riepilogo Struttura
- `/admin`: Dashboard di orchestrazione.
- `/public`: Front-end visibile agli utenti.
- `/data`: Database JSON.
- `server.py`: Motore di sincronizzazione locale.
- `handle_project.sh`: Gestore dei comandi di vita del progetto.
